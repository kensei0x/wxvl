最近的漏洞挖掘过程中，我有一个非常直观的感受：**现在的攻防环境，和两三年前相比已经发生了质的变化。**  
  
排除在客户现场进行内网渗透的情况，现在无论是打 SRC 还是做外网的渗透测试，只要发起大面积的扫描行为（无论是使用扫描插件还是传统的漏扫工具），绝大多数情况下面临的都是秒封。这说明现在的企业，哪怕是中小企业，其云服务器通常都配备了云 WAF，本地机房也多多少少部署了安全防护设备。  
  
此外，在日常的互联网业务渗透中，非常不建议使用“漏扫 + 高并发”的粗暴方式去探测资产。一旦把客户的业务系统打挂了，引发业务中断，后果是不堪设想的。  
  
**那么，在不能依赖大规模扫描的情况下，我们该怎么挖洞？**  
  
我通常将漏洞简单分为两类进行关注：**代码缺陷漏洞**  
 和 **逻辑缺陷漏洞**  
。  
1. **代码缺陷漏洞（如 SQL 注入、文件上传等）：**  
这是大部分新手接触安全的“第一课”。但新手的通病往往是：懂漏洞的定义，却不理解背后的代码执行逻辑，换个环境就不会挖了。  
  
1. SQL注入：  
通常在登录、注册、留言、搜索等参数交互点测试。  
  
1. 任意文件上传：  
头像修改、附件上传、文章配图等点位。  
  
1. RCE：  
除了利用框架已知的历史漏洞，绝大部分深层漏洞依然需要依赖代码审计。  
  
1. **逻辑缺陷漏洞：**  
比如在一个登录页面，我们抓包修改响应码看能否绕过前端校验；寻找隐藏的注册入口；对 API 接口进行参数 FUZZ；测试验证码复用；或者在“忘记密码”流程中寻找重置凭证的缺陷。可以参考我之前的一篇文章：[漏洞挖掘|EDUSRC高危漏洞之巧妙拿下学校门户系统管理员权限](https://mp.weixin.qq.com/s?__biz=MzkzMDE5OTQyNQ==&mid=2247484965&idx=1&sn=31aba139fb7b4ccf33576bc4a0f9fc89&scene=21#wechat_redirect)  
  
  
对于经验丰富的师傅来说，他们可能凭借直觉和插件就能嗅到哪里存在 SQL 注入，如果被拦截再去研究如何绕过 WAF。而对我个人而言，我目前更喜欢挖掘**逻辑漏洞**  
。  
  
原因有两点：  
  
第一，**足够隐蔽**  
，正常的业务逻辑请求几乎不会触发安全设备的告警；  
  
第二，**危害往往直达核心**  
，当我们利用逻辑漏洞获取到一定的数据或权限后，往往能以此为跳板，继续深挖后台潜藏的代码缺陷漏洞。

### 深入挖掘：从账号 ID 到交付物品 ID 的敏感信息泄露  
  
顺着这个思路，我进入了“个人合同”功能页。  
  
![](https://mmbiz.qpic.cn/sz_mmbiz_png/mOj9kVLGEa0EsWyeCl9M2llN7BrACDDDR2cH9oreJrxyD7hhvAfv3L3x0OxfSQOiabpVKeeCaDDKV4XomuhfjuKE2vg1YKHw7QzAAOt4kBBY/640?wx_fmt=png&from=appmsg "")  
  
个人合同 API 请求包与响应包，包含交付物品 id、位置及合同时间  
  
还记得我们刚登录时，响应包返回的那个 id:  
1249  
吗？在请求合同列表的 API 中，URL 里恰好带了这个  
_id=1249  
参数。  
  
这其实是一个典型的后端逻辑缺失。  
- **前端行为：**  
将本地存储的账号 ID 拼接到 URL 发起请求。  
  
- **后端行为：**  
接收到 _id 参数后，直接带入类似  
SELECT * FROM contracts WHERE account_id = #{_id}  
的 SQL 语句中查询，而**完全没有去校验当前请求头中的****User-Token****是否真正属于这个****_id**  
。  
  
我尝试将 URL 中的 _id 修改为其他数字，成功越权获取到了他人的合同概览信息，包括具体的物理位置和合同到期时间。

### 案例一总结与思考  
  
这个案例是一个典型的**多重平行越权组合**  
：  
1. **通过****userId**  
：越权执行了他人账号的应用功能（如签到）。  
  
1. **通过账号****id**  
：越权查询到了他人的资产绑定概览。  
  
1. **通过交付物品****id**  
：越权获取了最核心的个人身份合同原件。  
  
**问题根源（技术视角）：**  
后端在设计 API 时，过度信任了前端传来的业务标识符（ID）。正确的做法应该是在后端建立一套完整的会话上下文校验。  
- **修复建议：**  
 任何涉及数据查询的接口，必须将传入的参数与当前  
User-Token  
解析出的用户身份进行强关联校验。

Proxifier

BurpSuite

### 案例二总结与思考  
  
虽然该系统在权限校验（越权拦截）和 JWT 安全上做得不错，但在支付逻辑上犯了一个致命的错误。  
  
**问题根源（技术视角）：**  
后端将“业务订单”和“支付订单”的校验逻辑分离了，且未做最终对账。  
- 前端传了业务金额（XXMoney）和支付金额（money）。  
  
- 后端用 money 生成了微信支付单据。  
  
- 微信回调成功后，后端仅仅判断了“支付动作已完成”，然后就**盲目采信了请求包中的****XXMoney**  
 作为服务下发的依据，并没有去比对微信实际收到的钱和业务所需扣除的钱是否一致。  
  
**修复建议：**  
取消前端对支付金额的控制权。前端只需传入“商品/服务 ID”即可，后端根据 ID 查库计算金额，生成支付单。微信回调后，不仅要校验支付状态，还必须强制校验 Total_Fee（实际支付金额）是否大于等于该订单在数据库中记录的应付金额。  
## 案例三：某服务行业子业务单位资产的 API 未授权访问  
  
在针对大型目标（资产多、分支机构多）进行渗透时，前期信息收集的广度和深度，直接决定了后续漏洞产出的质量。  
  
给大家分享一下我的信息收集思路：  
1. **股权穿透：**  
确定目标主单位及其控股的下属单位。  
  
1. **AI****辅助：**  
合理利用大模型提取和整理企业关联信息（提示词要写好）。  
  
1. **备案查询：**  
利用 ICP 域名备案系统，准确查询目标名下的网站、APP 和小程序（第三方数据库往往存在滞后性）。  
  
1. **空间测绘：**  
综合使用 Hunter、Quake、Fofa 等平台进行语法检索。  
  
1. **交叉存活探测：**  
导出资产列表，使用 HTTPX 等工具批量探测存活，为后续漏洞挖掘提供标靶。

### 案例三总结与思考  
  
**问题根源（技术视角）：**  
典型的 API 路由鉴权缺失与敏感数据过度暴露。  
- 在 Spring Boot 等后端框架中，通常会使用拦截器（@Interceptor）或鉴权框架（如 Shiro、Spring Security）来保护 /api/** 路径。但开发人员可能在配置路由白名单时出现了疏漏，导致 userlist 接口裸露。  
  
- **掩耳盗铃的数据加密：**  
试图通过 Hex.encode() 将数据混淆，这在渗透人员眼里形同虚设。  
  
- **数据脱敏缺失：**  
即使是合法的后台管理员请求用户列表，后端也绝对不应该将用户的 Password 字段（哪怕是 Hash 值）下发给前端。应使用 DTO（数据传输对象）过滤掉不必要的敏感字段。

### 案例四总结与思考  
  
这是一个非常经典的因**架构简陋与多处低危逻辑缺陷串联**  
，最终导致高危系统接管的案例。  
1. **任意手机号模糊查询（低危）：**  
接口缺乏参数复杂度校验，导致全表数据遍历泄露。  
  
1. **越权信息查询（中危）：**  
利用已知标识符，越权获取敏感手机号。  
  
1. **脆弱的认证机制（高危）：**  
仅凭手机号存在与否就放行登录，完全没有考虑身份伪造的风险。  
  
**修复建议：**  
- 废弃极其脆弱的“存在即登录”逻辑，必须引入短信验证码或密码校验。  
  
- 模糊查询接口必须限制最少输入字符长度（如至少输入 4 位手机号尾号），并限制返回结果条数（  
LIMIT  
）。  
  
## 最终总结：一点实战感想  
  
整理完这些案例，我也想和大家分享几点我个人的感悟。  
1. **思路决定上限，技术决定下限。**  
挖掘逻辑漏洞，最重要的不是手里掌握着多么牛的 0day 工具，而是**对业务的理解能力**  
。拿到一个系统，需要像产品经理一样去思考：这个功能是给谁用的？数据是怎么流转的？前端发了什么，后端又盲目相信了什么？很多时候，逻辑漏洞就藏在那些看似合理，实则缺乏二次校验的业务缝隙里。  
  
1. **经验是真刀真枪打出来的。**  
同样的系统，有的师傅一登录，随便点几下，脑子里就已经浮现出三四条攻击路径，优先测什么，后手备什么，门儿清。这是为什么？这不是看看视频、背背文章就能具备的能力，这是在成百上千个实战项目中磨出来的嗅觉。所以，一定要多上手去实践。  
  
1. **“黑盒测试，白盒思维”。**  
在做外网渗透时，我们往往面对的是一个黑盒。但优秀的渗透测试人员，能够通过抓包和报错信息，在大脑里逆向“还原”出后端大概的代码逻辑和 SQL 语句。只有当你“看到”了代码是怎么写的，你才知道怎么去构造 Payload 打穿它。  
  
渗透测试就像是在和开发人员进行一场跨越时空的心理博弈。不要惧怕被 WAF 拦截，沉下心来，理清逻辑，总能找到破局的那个点。
